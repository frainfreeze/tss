1 # tss - Tom's Simple Shell
2 A shell or command-line interpreter is a user interface for access to an operating system's
3 services. In general, operating system shells use either a command-line interface (CLI)
4 or graphical user interface (GUI), depending on a computer's role and particular
5 operation. It is named a shell because it is the outermost layer around the operating
6 system kernel.

8 > Although most users think of the shell as an interactive command interpreter, it is really a programming language in which each statement runs a command. Because it must satisfy both the interactive and programming aspects of command execution, it is a strange language, shaped as much by history as by design. — Brian Kernighan & Rob Pike

10 ## Shell in a nutshell
11 In it's essence shell is a Read–Eval–Print Loop (REPL), a simple, interactive computer
12 programming environment that takes single user inputs (i.e. single expressions), evaluates
13 them, and returns the result to the user.
14 
15 Typical shell workflow is as follows:
16 - Startup: read and execute configuration files
17 - Wait for user input and interpret (read commands from stdin (interactive/file) and execute them)
18  - Parsing
19  - Expansion
20  - Execution
21 - Terminate: executes any shutdown commands, free up any memory, and terminate

27 ## System calls
28 The way our programs talk to the kernel is through system calls.
29 These are operations we can tell the kernel to do on our behalf.
30 Examples include:
31  - opening files (`open`)
32  - reading and writing to them (`read`, `write`)
33  - sending and receiving data over the network. (`sendto`, `recvfrom`)
34  - starting programs (`execve`)
35  - process creating a copy of itself (`fork`)
36 
37 In fact, more or less everything that involves our programs interacting
38 with the ‘real world’ involves system calls.
39 
40 Shell is parent process so it utilizes system call `fork`, this helps us
41 to isolate commands which is important for fault tolerance. We dont want
42 erroneous command to crash whole shell. The `fork` creates a new process
43 that is a clone of the existing one, right down to the point in the program
44 that's executing. We can then call exec in the newly created child process,
45 leaving our shell's process around for future interaction.
46 
47 The return value of fork tells us if we're in the child or the parent:
48 it returns 0 for the child, and the id of the new process in the parent.
49 
50 To wait for the child process to finish we can use `wait` system call.

52 ## Builtins
53 Every process has its own its own memory and extra metadata that the kernel
54 manages, but that the process can't change directly. Process can use system
55 calls to change user id (`setuid`), group id (`setgid`), priority (`setpriority`),
56 working directory (`chdir`).
57 
58 If you were to call change directory command it would invoke `chdir` internally,
59 change its working dir and return to the parent process - however parents process
60 working dir would stay unchanged, therefore shell has to have builting `cd` command
61 that calls `chdir` system call on shell process itself. The handling of cd is then
62 built in to our shell — it's a builtin.
63 
64 In bash, `export` lets us change an environment variable's value in the shell itself.
65 As with `cd`, because a child cannot modify its parent's environment, export must be
66 a builtin.
67 
68 See manpages on bash builtins [man builtins](https://linux.die.net/man/1/builtins) for more.

70 ## Pipeline
71 In Unix-like computer operating systems, a pipeline is a sequence of processes
72 chained together by their standard streams, so that the output of each process
73 (stdout) feeds directly as input (stdin) to the next one.
74 
75 Pipes are unidirectional; data flows through the pipeline from left to right.
76 
77 ```
78 process1 | process2 | process3
79 ```
80 
81 The `pipe()` system call asks the operating system to construct a new anonymous
82 pipe object. This results in two new, opened file descriptors in the process:
83 the read-only end of the pipe, and the write-only end. The pipe ends appear to
84 be normal, anonymous file descriptors, except that they have no ability to seek.
85 
86 To avoid deadlock and exploit parallelism, the Unix process with one or more new
87 pipes will then, generally, call `fork()` to create new processes. Each process
88 will then close the end(s) of the pipe that it will not be using before producing
89 or consuming any data. Alternatively, a process might create a new thread and use
90 the pipe to communicate between them.
91 
92 Named pipes may also be created using `mkfifo()` or `mknod()` and then presented
93 as the input or output file to programs as they are invoked. They allow multi-path
94 pipes to be created, and are especially effective when combined with standard error
95 redirection, or with tee.
96 
97 In most Unix-like systems, all processes of a pipeline are started at the same time,
98 with their streams appropriately connected, and managed by the scheduler together with
99 all other processes running on the machine. An important aspect of this, setting Unix
100 pipes apart from other pipe implementations, is the concept of buffering: for example
101 a sending program may produce 5000 bytes per second, and a receiving program may only
102 be able to accept 100 bytes per second, but no data is lost. Instead, the output of the
103 sending program is held in the buffer. When the receiving program is ready to read data,
104 then next program in the pipeline reads from the buffer. In Linux, the size of the buffer
105 is 65536 bytes (64KiB).

113 sends a signal to your process, in this case `SIGINT`. Signals interrupt the normal execution of

124 ### Running a command in the background
125 In your regular shell, if you end a command with &, it runs in the background. The command will
